- DB에서 테이블을 CREATE 하는데 자동으로 PK 관련 INDEX가 하나 생성된다. 이후 자주 쓰일 것 같은 조건에 관련한 INDEX 작업을 진행해 놓아야 쿼리 성능이 증가한다.
- 단 INDEX 자체도 어떻게 미리 SORTING 한 가상의 데이터를 가져다 놓는 방식이기 때문에 과도하게 설정 시 DB에 무리가 간다.

```SQL
SELECT *
FROM TB_TABLE1
WHERE COL1=
AND   COL2 = 
AND   COL3 BETWEEN A AND B
AND   COL4 LIKE '~%'
```

위와 같이 형성이 되는데 이때 주로 날짜 관련된 데이터를 사용해 조회가 된다면 COL3 으로 인덱스 처리를 미리해놓는게 좋다.

만약 검색조건에서 무조건 COL1, COL2 가 들어간다면 추가하는게 좋다. 인덱스는 차례대로 정렬해놓는 시스템이기 때문에 만약 인덱스를
COL1, COL2, COL3 으로 설정해 놓고 COL1 이 조회조건에서 제외된다면 해당 조회SQL은 FULL SCAN으로 성능이 저하될 수 밖에 없다. 

따라서 인덱스를 설정할때 가장 중요한점은 해당 검색을 사용할 때 무조건 있는 조건이 앞쪽 없을 수도 있는 조건이 뒤로 가야한다.

위의 경우에 COL1은 가끔씩 조건에서 빠지는 쿼리가 있고 COL2는 필수 , COL3도 조회에서 자주 빠진다면

INDEX는 COL2 , COL1, COL3 방식으로 설정되어야한다.

두번째로 인덱스에서 중요한점은 타깃되는 개수이다.
예를들어 COL2가 설정이 되더라도 대부분이 대상이라면 따로 인덱스를 통해 크게 성능을 개선하기는 힘들다.(예를들어 BOOLEAN 데이터인데 대부분이 TRUE라면 TRUE로 검색하는게 따로 의미 없다.)

다만 예를들어 택배시스템에서 대부분의 데이터는 이미 완료된 배송완료 데이터이다. 배송중인 데이터만 조회할 때는 배송완료 구분자를 두어 배송중인 데이터만 조회하는 서비스에서는 효과가 꽤 있을 것이다.

마지막으로 SORTING 기능이다. INDEX를 지정할 때 해당 인덱스는 ASC, DESC 를 지정한다. 예를들어 위 쿼리를 검색이후 COL3, COL4, COL5로 MULTI SORTING이 필요하다고 하자
이럴 때 인덱스를 COL3, COL4, COL5로 지정해놓으면 COL5는 조건에 없더라도 SELECT 되는 결과값이 COL3,COL4, COL5로 SORTING 되어서 나오는걸 알 수 있다.
