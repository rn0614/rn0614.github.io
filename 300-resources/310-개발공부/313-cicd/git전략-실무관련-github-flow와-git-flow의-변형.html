Git전략 실무관련 GITHUB FLOW와 GIT FLOW의 변형

개요

GIT FLOW는 일정주기로 배포가 되며, GITHUB FLOW는 배포 관련 검증 부분이 들어가 있지 않다.
실무에서는 develop 브랜치를 검증 공간으로 두고, feature를 직접 main에 반영하는 방식을 구현했다. 이는 Trunk-Based Development에 QA 검증 단계를 추가한 형태이다.

브랜치 관리 방안

  1. git-flow, github-flow 둘다 main은 안정적인 서비스 상태를 유지.
  2. github-flow의 경우 간단한 develop 없이 바로 main과 feature 상태에서만 유지하는 방식
  3. release와 feature은 merge 완료시 삭제하여 깨끗한 환경을 유지한다.

git flow 관련 브랜치 및 핵심원칙

브랜치 용도 생명주기
main 프로덕션 코드 영구
release/ 반영될 코드 준비 임시
develop 개발환경 코드 영구
feature/ 기능 개발 임시
hotfix/ 긴급 수정 임시 
main (항상 안정적)
└── 완료되고 검증된 것만

## release (develop에서 배포할 대상만 뽑은 형태)

develop (통합 테스트)
└── 기능들 합쳐서 테스트

feature/* (독립 개발)
├── feature/login (김개발)
├── feature/payment (이개발)
├── feature/ui (박개발)
└── feature/db (최개발)

핵심 원칙

  1. main은 항상 배포 가능해야 함
  2. feature는 main에서 분기 (develop에서 X)
  3. develop은 검증 전용 공간 (지저분해도 OK)
  4. main 배포 전 develop 검증 필수
  5. 기능별로 브랜치 분리
  6. 명확한 COMMIT MESSAGE규칙

문제상황 : 특이한 개발환경 내용

위의 방식에서 문제가 생겼다. 팀원이 대략 20명이고 20명은 하나의 프로젝트에 대해서만 반영하지 않고 실시간으로 문제를 개선한다.
사실상 hotfix 가 수차례 이뤄지는 환경인데 골때리는 것이 또 테스트는 필요한 상황이다. 즉, 프로젝트 자체는 mono하게 관리하고 있으나 시스템 자체의 결합구조가 크고 서로 영향이 없는 개발도 수십건이어서 모든 개발내용을 develop에 뭉쳐서 다음주에 반영한다는 것은 말이 되지 않았다.

이런 문제를 해소하기 위해 새로운 방식이 필요했다.

일반 환경과 다른 특이한 점

  1. 프로젝트 전체가 하나의 repository에 들어있다. 서로 연관은 있지만 독립적인 부분도 있다.
  2. 같이 프로그램을 유지보수 하는 개발자가 많다. 즉, 모두의 일정을 맞출 수 없다.
  3. 고객의 물리적인 일정도 하나의 날짜에 맞출 수 없는 상황
  4. 개발한 내용의 검증이 가능한 개발계, 검증계 환경은 구축된 상태

해결방안

feature 브랜치를 develop에 올리고 이를 release로 새 브랜치를 만들어 main에 merge 하는 방식이 아닌

feature 브랜치를 직접적으로 main으로 올리는 방식을 택했다. 다만 develop 브랜치를 QA의 장으로 두고 develop 브랜치에 올리지 않은 내용은 release, main으로 올리지 못하도록 하였다.

개선된 GIT Flow 시각화

  1. 모든 feature는 main으로부터 분기 bash git checkout main git checkout -b feature/xxx
  2. 2단계 검증 프로세스 - 1단계: feature → develop (QA 검증) - 2단계: feature → main (배포)
  3. develop은 검증 전용 공간 - 지저분해도 괜찮음 - 미완성 feature 포함 가능 - 주기적으로 삭제/재생성으로 정리
  4. ⚠️ 중요: 내용 동일성 검증 - main에 반영하는 feature 내용은 develop에서 테스트한 마지막 내용과 완전히 동일해야 함 - 자동 검증으로 확인

사용 명령어 정리

develop 배포 후 main 배포 시 검증 로직

Verify content identity 를 통해서 develop 을 merge 했을 때 코드와 지금 올리는 코드와 동일해야함. 즉 develop 개발 배포 코드를 한번이라도 탔어야함.

# .github/workflows/pr-to-main.yml
name: PR to Main - Auto Verification & CI

on:
  pull_request:
    branches: [main]

jobs:
  # 자동 검증
  auto-verification:
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v3
        with:
          fetch-depth: 0  # 전체 히스토리 필요
      
      - name: Fetch branches
        run: |
          git fetch origin develop:develop
          git fetch origin main:main
      
      - name: Verify content identity
        run: |
          BRANCH="${{ github.head_ref }}"
          
          echo "🔍 Checking content identity..."
          
          DEVELOP_MERGE=$(git log develop --grep="Merge.*$BRANCH" -1 --format="%H")
          DEVELOP_VERSION=$(git log $DEVELOP_MERGE^2 -1 --format="%H")
          CURRENT_VERSION=$(git log origin/$BRANCH -1 --format="%H")
          
          echo "Develop version: $DEVELOP_VERSION"
          echo "Current version: $CURRENT_VERSION"
          
          if [ "$DEVELOP_VERSION" != "$CURRENT_VERSION" ]; then
            echo "::error::❌ develop merge 이후 코드가 변경되었습니다!"
            echo ""
            echo "develop에 merge한 코드와 다릅니다."
            echo "변경사항이 있다면 develop에 다시 merge하고 재검증 받아야 합니다."
            exit 1
          fi
          
          echo "✅ 내용 일치"
      
      - name: Check verification time
        run: |
          BRANCH="${{ github.head_ref }}"
          
          MERGE_TIME=$(git log develop --grep="Merge.*$BRANCH" -1 --format="%ct")
          CURRENT_TIME=$(date +%s)
          
          MIN_WAIT=86400  # 1일 = 86400초
          ELAPSED=$((CURRENT_TIME - MERGE_TIME))
          HOURS=$((ELAPSED / 3600))
          
          if [ $ELAPSED -lt $MIN_WAIT ]; then
            REMAINING=$(((MIN_WAIT - ELAPSED) / 3600))
            
            echo "::warning::⚠️ develop merge 후 1일이 경과하지 않았습니다"
            echo "경과 시간: ${HOURS}시간"
            echo "남은 시간: ${REMAINING}시간"
            echo ""
            echo "권장: QA 검증 기간을 충분히 가진 후 merge하세요."
          else
            echo "✅ 검증 기간 충족: ${HOURS}시간"
          fi
      
      - name: Comment verification result
        uses: actions/github-script@v6
        if: always()
        with:
          script: |
            const mergeCount = '${{ steps.check_develop.outputs.merge_count }}';
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🔍 자동 검증 결과
              
✅ develop 선행 merge: 확인됨 (${mergeCount}회)
✅ 내용 동일성: 확인됨
⏱️ 검증 시간: 확인 중

자동 검증을 통과했습니다!`
            });

  # 코드 리뷰 및 CI 테스트
  code-review-and-ci:
    needs: auto-verification
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v3
      
      - name: Setup Node.js
        uses: actions/setup-node@v3
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      # 전체 테스트 재실행
      - name: Lint
        run: npm run lint
      
      - name: Format check
        run: npm run format:check
      
      - name: Type check
        run: npm run type-check
      
      - name: Unit tests
        run: npm run test:unit
      
      - name: Integration tests
        run: npm run test:integration
      
      - name: E2E tests
        run: npm run test:e2e
      
      - name: Security audit
        run: npm audit --audit-level=moderate
      
      - name: Build
        run: npm run build
        env:
          NODE_ENV: production
      
      - name: Upload build artifact
        uses: actions/upload-artifact@v3
        with:
          name: production-build
          path: dist/
          retention-days: 7
      
      - name: Comment test result
        uses: actions/github-script@v6
        if: always()
        with:
          script: |
            const status = '${{ job.status }}';
            const emoji = status === 'success' ? '✅' : '❌';
            
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `${emoji} 코드 리뷰 및 CI 테스트 ${status === 'success' ? '통과' : '실패'}`
            });
