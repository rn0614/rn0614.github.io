프론트엔드 배포 방식의 진화와 비용 최적화 전략

배경

프론트엔드 개발자 관점에서 서버 비용 최적화와 배포 전략에 대한 논의. SSR, CSR의 비용 구조를 이해하고, Next.js를 활용한 하이브리드 접근법과 다양한 배포 환경(EC2, Amplify, Vercel)의 비용 차이를 분석.

핵심 인사이트

1. 비용 최적화의 핵심은 백엔드 부하 감소

백엔드 비용이 대부분을 차지하는 이유

  • DB는 부하에 매우 취약한 구조
  • API 호출 1회 줄이는 것이 렌더링 최적화보다 비용 효과 큼
  • 프론트엔드 최적화 = 백엔드를 덜 건드리게 만드는 것

측정 지표

  • API 호출 수 (CloudWatch, DataDog)
  • DB CPU/메모리 사용률
  • 캐시 히트율

2. 배포 방식의 진화와 비용 구조

1세대: 전통적 SSR (2000년대)

기술: PHP, JSP, Ruby on Rails
배포: EC2 또는 온프레미스 서버

[클라이언트] → [로드밸런서] → [EC2] → [DB]

비용 구조:
- EC2 t3.medium: 월 $30-50 (24시간 고정 비용)
- 유휴 시간에도 과금
- 트래픽 증가 시 인스턴스 선형 증가

문제점:
- 서버 리소스 낭비
- 페이지 이동마다 전체 HTML 새로 받음
- 프론트/백엔드 역할 분리 어려움

2세대: CSR (2010년대 중후반)

기술: React, Vue, Angular
배포: S3 + CloudFront

[클라이언트] → [CloudFront] → [S3]
                    ↓
          필요시 [API 서버]

비용 구조:
- S3: $0.023/GB (거의 무시 가능)
- CloudFront: $0.085/GB
- 월 100만 페이지뷰: $10-20
- EC2 대비 90% 비용 절감

장점:
- 서버 비용을 클라이언트로 전가
- SPA로 부드러운 UX
- 명확한 프론트/백엔드 분리

단점:
- SEO 취약 (빈 HTML)
- 초기 로딩 느림 (JS 번들)
- 저사양 기기 부담

3세대: Hybrid (2020년대)

기술: Next.js, Nuxt, Remix
배포: SSR + CSR + SSG + ISR 혼용

페이지별 최적화:
- 마케팅/블로그 → SSG (CDN, API 호출 제로)
- 상품 목록 → ISR (주기적 재생성)
- 대시보드 → SSR (실시간)
- API Routes → Serverless

3. Next.js 배포 방식 비교

A. EC2 배포 (전통 방식)

구성:
[클라이언트] → [LB] → [EC2 Node.js] → [DB]

비용: $30-50/월 (고정)

장점:
- 기존 인프라 통합 용이
- VPC 프라이빗 리소스 접근
- 예측 가능한 비용

단점:
- 트래픽 변동 대응 비효율
- 스케일링 수동 관리

B. AWS Amplify 배포

구성:
[클라이언트] → [CloudFront] → SSG/ISR: S3
                            ↘ SSR: Lambda@Edge
                            ↘ API: Lambda

비용 (월 100만 요청):
- SSG/ISR: 거의 무료
- SSR Lambda@Edge: $6.25
- API Lambda: $0.20
→ EC2 대비 80-90% 절감

내부 동작:
pages/index.js (SSG) → S3 HTML 저장
pages/blog/[id].js (ISR) → S3 + 주기적 재생성
pages/api/user.js → Lambda 함수
pages/dashboard.js (SSR) → Lambda@Edge 렌더링

C. Vercel 배포 (최적화)

구성:
[클라이언트] → [Edge Network] → SSG: Edge CDN
                              ↘ SSR: Edge Functions
                              ↘ API: Serverless

비용:
- Free Tier: 100GB 대역폭, 100시간 빌드
- Pro: $20/월 (1TB 대역폭)

장점:
- Next.js 제작사의 최적화
- 전 세계 Edge 서버 SSR
- 자동 스케일링

프론트엔드 비용 최적화 전략

1. API 호출 자체를 줄이기

TanStack Query 활용

// 중복 요청 자동 제거
useQuery(['user', id], fetchUser)  // 실제 API 호출
useQuery(['user', id], fetchUser)  // 캐시에서 반환

// staleTime 설정으로 재요청 방지
useQuery(['posts'], fetchPosts, {
  staleTime: 5 * 60 * 1000  // 5분간 fresh
})

효과: API 호출 30-50% 감소

2. localStorage 전략적 사용

적용 대상

  • 코드성 데이터 (국가 목록, 카테고리)
  • 세션 유지 데이터 (폼 입력, 필터 설정)
  • 최근 검색어
// 첫 로드 후 localStorage 저장
const categories = await fetchCategories()
localStorage.setItem('categories', JSON.stringify(categories))

// 이후 API 호출 없이 사용
const cached = localStorage.getItem('categories')

// zustand-persist로 상태 영속화
const useStore = create(
  persist(
    (set) => ({ filters: {} }),
    { name: 'user-filters' }
  )
)

효과: 
- 백엔드 요청 제로
- 모바일 네트워크 레이턴시 개선

3. 상태관리 최적화

Context API의 문제

// 상위 Context 변경 시 하위 전체 리렌더링
// → 불필요한 연산 → 성능 저하

Zustand 도입 기준

  • 컴포넌트 트리가 깊음
  • 업데이트가 빈번함
  • 부분 리렌더링 필요
// 필요한 컴포넌트만 리렌더링
const bears = useStore((state) => state.bears)

트레이드오프:
- 번들 사이즈 증가 (~3KB)
- 리렌더링 비용 감소

4. 낙관적 업데이트

// 좋아요 버튼 예시
onClick: () => {
  // 1. 즉시 UI 반영
  setLikes(prev => prev + 1)
  
  // 2. 백그라운드 API 호출
  mutate({ postId })
  
  // 3. 실패 시 롤백
}

효과:
- 체감 성능 대폭 개선
- 전환율 증가
- 백엔드 부하는 동일

5. Next.js 페이지별 최적화

// next.config.js
export default {
  // 마케팅 페이지 → SSG
  '/': { revalidate: 3600 },
  '/about': { revalidate: false },
  
  // 상품 목록 → ISR
  '/products': { revalidate: 600 },  // 10분
  
  // 대시보드 → SSR
  '/dashboard': 'ssr',
  
  // API → Serverless
  '/api/*': 'serverless'
}

비용 효과 (100개 페이지):
- 70개 SSG → $0
- 20개 ISR → $2
- 10개 SSR → $8
총 $10 vs EC2 $50

실전 비용 비교

방식 구성 월 비용 (100만 방문) 특징
전통 SSR EC2 t3.medium $50 24시간 고정
CSR S3 + CloudFront $15 API 별도
Next.js + EC2 EC2 $50 유연성 낮음
Next.js + Amplify Lambda + S3 $10-30 트래픽 비례
Next.js + Vercel Edge Functions $0-20 최적 성능

의사결정 가이드

EC2 선택 시

  • 기존 인프라 통합 필요
  • VPC 프라이빗 리소스 접근
  • 24시간 고트래픽 (비용 효율 역전)
  • 특수 런타임 환경

Amplify/Vercel 선택 시

  • 트래픽 변동 큼 (피크 타임)
  • 글로벌 서비스 (Edge)
  • 빠른 배포, 자동 스케일링
  • 비용 최적화 우선

하이브리드 전략

- 메인 웹: Vercel (Next.js)
- 관리자: EC2 (보안)
- API: Lambda (트래픽 비례)

최적화 시점 기준

  • DAU 10만+ 또는 월 API 호출 1억+
  • 서버 비용 월 $1000 이상
  • 모바일 트래픽 50% 이상

결론

  1. CSR의 비용 혁명: 서버 비용을 클라이언트로 전가하여 90% 절감
  2. Hybrid의 균형: Next.js로 SEO + 비용 최적화 동시 달성
  3. 배포 환경이 비용 결정: 같은 Next.js도 배포 방식에 따라 5배 차이
  4. 프론트 최적화 = 백엔드 부하 감소: API 호출 줄이기가 핵심
  5. Serverless가 현대 표준: Amplify/Vercel로 트래픽 변동 대응
