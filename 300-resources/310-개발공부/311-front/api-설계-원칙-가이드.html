API 설계 원칙 가이드

RESTFUL 원칙 가이드

원칙 설명 REST API 적용 예시
리소스 지향 설계 리소스기반, 명사 사용 GET /users, POST /articles, DELETE /comments/123
HTTP 메서드 활용 각 메서드의 의미를 정확히 사용 GET(조회), POST(생성), PUT(수정), DELETE(삭제)
계층적 URI 구조 리소스 간 관계를 URI로 표현 /users/123/orders/456 (123번 사용자의 456번 주문)
복수형 명사 사용 컬렉션은 복수형으로 일관되게 표현 /users (O), /user (X)
상태 코드 활용 적절한 HTTP 상태 코드로 결과 전달 200(성공), 201(생성), 400(잘못된 요청), 404(없음), 500(서버 오류)
필터링/정렬/페이징 쿼리 파라미터로 데이터 제어 GET /users?page=2&limit=20&sort=created_at&status=active
버전 관리 API 버전을 명시적으로 관리 /v1/users, /v2/users 또는 Accept: application/vnd.api.v1+json
HATEOAS 응답에 관련 링크 포함 (선택적) { "id": 1, "links": { "self": "/users/1", "orders": "/users/1/orders" }}
무상태성 각 요청은 독립적, 서버는 세션 저장 안 함 인증 토큰을 매 요청마다 헤더에 포함 (Authorization: Bearer token)
일관된 응답 구조 성공/실패 응답 형식 통일 { "data": {...}, "meta": {...}, "errors": [...] }
에러 처리 명확하고 도움이 되는 에러 메시지 { "error": { "code": "INVALID_EMAIL", "message": "이메일 형식이 올바르지 않습니다" }}
보안 HTTPS, 인증/인가, Rate Limiting Authorization: Bearer eyJhbG..., X-RateLimit-Remaining: 95
캐싱 HTTP 캐싱 헤더 활용 Cache-Control: max-age=3600, ETag: "33a64df551425fcc"
멱등성 보장 PUT, DELETE는 여러 번 호출해도 같은 결과 PUT /users/123 여러 번 호출 → 같은 상태 유지
Content Negotiation 클라이언트가 원하는 형식으로 응답 Accept: application/json 또는 Accept: application/xml
