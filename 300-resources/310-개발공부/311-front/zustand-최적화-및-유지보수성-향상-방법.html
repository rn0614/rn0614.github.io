Zustand 최적화 및 유지보수성 향상 방법

개요

Zustand를 최적화/유지보수하기 좋게 사용하는 법

핵심 원칙

기본내용

  1. 선택적 구독으로 불필요한 리렌더링 방지
  2. shallow 비교로 객체 상태 구독하기
  3. actions 객체로 액션 관리
  4. DevTools로 상태 변화 추적

추가내용

  1. Persist로 새로고침 후에도 상태 유지
  2. 파생 상태는 컴포넌트에서 처리
  3. Subscribe로 React 외부에서 접근
  4. 비동기 액션의 loading/error 상태 관리

1. 선택적 구독으로 불필요한 리렌더링 방지

하나의 store에서 여러 상태를 관리하더라도 구독한 상태만 변경될 때만 리렌더링된다.

const useStore = create((set) => ({
  user: { name: 'John', age: 30 },
  count: 0,
  theme: 'dark',
  setUser: (user) => set({ user }),
  incrementCount: () => set((state) => ({ count: state.count + 1 })),
  setTheme: (theme) => set({ theme })
}))

// ❌ 나쁜 예: 전체 store 구독
const BadComponent = () => {
  const store = useStore() // 모든 상태 변경시 리렌더링!
  return <div>{store.user.name}</div>
}

// ✅ 좋은 예: 필요한 것만 선택적 구독
const GoodComponent = () => {
  const userName = useStore(state => state.user.name)
  return <div>{userName}</div>
}

핵심: count가 변해도 GoodComponent는 리렌더링 안됨!

2. shallow 비교로 객체 상태 구독하기

여러 상태를 객체로 가져올 때는 shallow 옵션을 사용하여 실제 값이 변경될 때만 리렌더링되도록 한다.

import { shallow } from 'zustand/shallow'

const useStore = create((set) => ({
  firstName: 'John',
  lastName: 'Doe',
  age: 30,
  theme: 'dark',
  setFirstName: (firstName) => set({ firstName }),
  setAge: (age) => set({ age })
}))

// ❌ 나쁜 예: 매번 새 객체 생성
const BadComponent = () => {
  const { firstName, lastName } = useStore(state => ({ 
    firstName: state.firstName,
    lastName: state.lastName
  })) // 다른 상태(age, theme) 변경시에도 리렌더링!
  
  return <div>{firstName} {lastName}</div>
}

// ✅ 좋은 예: shallow 비교
const GoodComponent = () => {
  const { firstName, lastName } = useStore(
    state => ({ 
      firstName: state.firstName, 
      lastName: state.lastName 
    }),
    shallow // 실제 firstName, lastName이 변경될 때만 리렌더링
  )
  
  return <div>{firstName} {lastName}</div>
}

3. actions 객체로 액션 관리

액션은 변하지 않는 함수이므로 actions 객체에 담아서 관리하면 상태와 구분이 명확하고 사용할 때도 편리하다.

// ❌ 나쁜 예: 상태와 액션이 섞여있음
const useStore = create((set) => ({
  count: 0,
  user: null,
  theme: 'light',
  increment: () => set((state) => ({ count: state.count + 1 })),
  decrement: () => set((state) => ({ count: state.count - 1 })),
  setUser: (user) => set({ user }),
  setTheme: (theme) => set({ theme })
}))

// ✅ 좋은 예: actions 객체로 분리
const useStore = create((set) => ({
  // 상태
  count: 0,
  user: null,
  theme: 'light',
  
  // 액션
  actions: {
    increment: () => set((state) => ({ count: state.count + 1 })),
    decrement: () => set((state) => ({ count: state.count - 1 })),
    setUser: (user) => set({ user }),
    setTheme: (theme) => set({ theme })
  }
}))

const Component = () => {
  const count = useStore(state => state.count)
  const { increment, decrement } = useStore(state => state.actions)
  
  return (
    <div>
      <p>{count}</p>
      <button onClick={increment}>+</button>
      <button onClick={decrement}>-</button>
    </div>
  )
}

더 나은 패턴: 커스텀 훅으로 분리

export const useCount = () => useStore(state => state.count)
export const useActions = () => useStore(state => state.actions)

4. DevTools로 상태 변화 추적

DevTools 미들웨어를 사용하면 Redux DevTools에서 상태 변화를 추적할 수 있다. useEffect의 기명함수처럼 개발 도구에서 어떤 액션이 실행됐는지 명확하게 확인할 수 있다.

import { devtools } from 'zustand/middleware'

const ACTION_TYPES = {
  INCREMENT: 'counter/increment',
  DECREMENT: 'counter/decrement',
  SET_USER: 'user/set'
}

const useStore = create(
  devtools(
    (set) => ({
      count: 0,
      user: null,
      
      actions: {
        increment: () => set(
          (state) => ({ count: state.count + 1 }), 
          false, 
          ACTION_TYPES.INCREMENT // DevTools에 표시될 액션 이름
        ),
        decrement: () => set(
          (state) => ({ count: state.count - 1 }), 
          false, 
          ACTION_TYPES.DECREMENT
        ),
        setUser: (user) => set({ user }, false, ACTION_TYPES.SET_USER)
      }
    }),
    { name: 'AppStore' } // DevTools에 표시될 Store 이름
  )
)

5. Persist로 새로고침 후에도 상태 유지

Persist 미들웨어를 사용하면 상태를 localStorage에 저장하여 새로고침 후에도 동일한 상태를 유지할 수 있다.

import { persist, createJSONStorage } from 'zustand/middleware'

// 기본 사용 - localStorage 자동 저장
const useUserStore = create(
  persist(
    (set) => ({
      user: null,
      theme: 'light',
      language: 'ko',
      
      actions: {
        setUser: (user) => set({ user }),
        setTheme: (theme) => set({ theme }),
        setLanguage: (language) => set({ language })
      }
    }),
    {
      name: 'user-storage' // localStorage key 이름
    }
  )
)

// 고급 옵션
const useSettingsStore = create(
  persist(
    (set) => ({
      theme: 'light',
      language: 'ko',
      notifications: true,
      
      actions: {
        setTheme: (theme) => set({ theme }),
        setLanguage: (language) => set({ language })
      }
    }),
    {
      name: 'app-settings',
      storage: createJSONStorage(() => sessionStorage), // sessionStorage 사용
      partialize: (state) => ({
        theme: state.theme,
        language: state.language
      }), // 특정 상태만 저장
      version: 1, // 버전 관리
      migrate: (persistedState, version) => {
        if (version === 0) {
          return { ...persistedState, notifications: true }
        }
        return persistedState
      }
    }
  )
)

6. 파생 상태는 컴포넌트에서 처리

파생 상태(filtered list, computed values)는 Store에서 관리하지 말고 필요한 원본 데이터를 가져와서 컴포넌트에서 처리하거나, selector 함수로 분리하는 것을 권장한다.

const useStore = create((set) => ({
  todos: [],
  filter: 'all', // 'all' | 'active' | 'completed'
  
  actions: {
    addTodo: (text) => set((state) => ({
      todos: [...state.todos, { 
        id: Date.now(), 
        text, 
        completed: false 
      }]
    })),
    setFilter: (filter) => set({ filter })
  }
}))

// ✅ 좋은 예 1: 컴포넌트에서 직접 처리 (권장)
const TodoList = () => {
  const { todos, filter } = useStore(
    state => ({ todos: state.todos, filter: state.filter }),
    shallow
  )
  
  // 컴포넌트에서 필터링
  const filteredTodos = todos.filter(todo => {
    if (filter === 'active') return !todo.completed
    if (filter === 'completed') return todo.completed
    return true
  })
  
  return (
    <ul>
      {filteredTodos.map(todo => (
        <li key={todo.id}>{todo.text}</li>
      ))}
    </ul>
  )
}

// ✅ 좋은 예 2: selector 함수로 분리
// store.js 파일 아래에 selector 함수들 정의
export const selectFilteredTodos = (state) => {
  const { todos, filter } = state
  return todos.filter(todo => {
    if (filter === 'active') return !todo.completed
    if (filter === 'completed') return todo.completed
    return true
  })
}

const BetterTodoList = () => {
  const filteredTodos = useStore(selectFilteredTodos)
  
  return (
    <ul>
      {filteredTodos.map(todo => (
        <li key={todo.id}>{todo.text}</li>
      ))}
    </ul>
  )
}

7. Subscribe로 React 외부에서 접근

subscribe 함수를 사용하면 React 컴포넌트 외부(utils, services 등)에서도 store의 상태 변화를 감지하고 처리할 수 있다.

const useStore = create((set) => ({
  user: null,
  isOnline: false,
  
  actions: {
    setUser: (user) => set({ user }),
    setOnline: (isOnline) => set({ isOnline })
  }
}))

// utils/analytics.js - React 외부에서 사용
export const setupAnalytics = () => {
  // 사용자 로그인 추적
  const unsubscribe = useStore.subscribe(
    (state) => state.user,
    (user, prevUser) => {
      if (user && !prevUser) {
        analytics.track('User Login', { userId: user.id })
      } else if (!user && prevUser) {
        analytics.track('User Logout', { userId: prevUser.id })
      }
    }
  )
  
  return unsubscribe
}

// services/websocket.js
export const setupWebSocket = () => {
  let ws = null
  
  const unsubscribe = useStore.subscribe(
    (state) => state.isOnline,
    (isOnline) => {
      if (isOnline && !ws) {
        ws = new WebSocket('wss://example.com')
      } else if (!isOnline && ws) {
        ws.close()
        ws = null
      }
    }
  )
  
  return () => {
    unsubscribe()
    ws?.close()
  }
}

8. 비동기 액션의 loading/error 상태 관리

API 호출시 loading과 error 상태를 함께 관리해야 한다. 더 효율적으로 하려면 react-query와 함께 사용하는 것을 추천한다.

// 기본 패턴: loading/error 직접 관리
const useProductStore = create((set) => ({
  products: [],
  isLoading: false,
  error: null,
  
  actions: {
    fetchProducts: async () => {
      set({ isLoading: true, error: null })
      
      try {
        const response = await fetch('/api/products')
        const products = await response.json()
        set({ products, isLoading: false })
      } catch (error) {
        set({ error: error.message, isLoading: false })
      }
    },
    
    deleteProduct: async (id) => {
      // Optimistic Update
      const currentProducts = useProductStore.getState().products
      set((state) => ({ 
        products: state.products.filter(p => p.id !== id) 
      }))
      
      try {
        await fetch(`/api/products/${id}`, { method: 'DELETE' })
      } catch (error) {
        // 실패시 롤백
        set({ products: currentProducts, error: error.message })
      }
    }
  }
}))

// 컴포넌트에서 사용
const ProductList = () => {
  const { products, isLoading, error } = useProductStore(
    state => ({
      products: state.products,
      isLoading: state.isLoading,
      error: state.error
    }),
    shallow
  )
  const { fetchProducts, deleteProduct } = useProductStore(
    state => state.actions
  )
  
  useEffect(() => {
    fetchProducts()
  }, [fetchProducts])
  
  if (isLoading) return <div>Loading...</div>
  if (error) return <div>Error: {error}</div>
  
  return (
    <ul>
      {products.map(product => (
        <li key={product.id}>
          {product.name}
          <button onClick={() => deleteProduct(product.id)}>Delete</button>
        </li>
      ))}
    </ul>
  )
}

ZUSTAND 참고 규칙

성능 최적화 규칙

규칙: 선택적 구독 사용

  • WHEN store에서 상태를 가져올 때
  • THEN 전체 store가 아닌 필요한 상태만 선택하여 구독
  • WHY 구독하지 않은 상태가 변경되어도 컴포넌트가 리렌더링되지 않음

CORRECT:

const userName = useStore(state => state.user.name)

INCORRECT:

const store = useStore() // 모든 상태 변경시 리렌더링

규칙: 객체 구독시 shallow 비교 필수

  • WHEN 여러 상태를 객체 형태로 구독할 때
  • THEN shallow 옵션을 반드시 추가
  • WHY shallow 없으면 매 렌더링마다 새 객체 생성으로 항상 리렌더링됨

CORRECT:

const { firstName, lastName } = useStore(
  state => ({ firstName: state.firstName, lastName: state.lastName }),
  shallow
)

INCORRECT:

const { firstName, lastName } = useStore(state => ({ 
  firstName: state.firstName, 
  lastName: state.lastName 
})) // shallow 없음

코드 구조 규칙

규칙: 상태와 액션 분리

  • WHEN store를 정의할 때
  • THEN 상태와 액션을 명확히 구분하고, 모든 액션은 actions 객체 안에 정의
  • WHY 상태와 액션이 구분되어 가독성과 유지보수성 향상

CORRECT:

const useStore = create((set) => ({
  // 상태
  count: 0,
  user: null,
  
  // 액션
  actions: {
    increment: () => set((state) => ({ count: state.count + 1 })),
    setUser: (user) => set({ user })
  }
}))

INCORRECT:

const useStore = create((set) => ({
  count: 0,
  user: null,
  increment: () => set((state) => ({ count: state.count + 1 })),
  setUser: (user) => set({ user })
}))

규칙: 커스텀 훅으로 분리

  • WHEN 특정 상태나 액션을 자주 사용할 때
  • THEN 커스텀 훅을 만들어 export
  • WHY 코드 재사용성과 명확성 향상

CORRECT:

export const useCount = () => useStore(state => state.count)
export const useActions = () => useStore(state => state.actions)

// 사용
const count = useCount()
const { increment } = useActions()

디버깅 규칙

규칙: DevTools 미들웨어 사용

  • WHEN store를 생성할 때
  • THEN devtools 미들웨어를 적용하고 모든 액션에 명확한 이름 지정
  • WHY Redux DevTools에서 상태 변화 추적이 용이하여 디버깅 효율 향상

CORRECT:

const ACTION_TYPES = {
  INCREMENT: 'counter/increment',
  DECREMENT: 'counter/decrement'
}

const useStore = create(
  devtools(
    (set) => ({
      count: 0,
      actions: {
        increment: () => set(
          (state) => ({ count: state.count + 1 }), 
          false, 
          ACTION_TYPES.INCREMENT
        )
      }
    }),
    { name: 'AppStore' }
  )
)

영속성 규칙

규칙: 새로고침 후 유지해야 하는 상태는 Persist 사용

  • WHEN 사용자 설정, 인증 정보 등 새로고침 후에도 유지해야 하는 상태가 있을 때
  • THEN persist 미들웨어 적용
  • WHY localStorage/sessionStorage에 자동 저장되어 새로고침 후에도 상태 유지

CORRECT:

const useUserStore = create(
  persist(
    (set) => ({
      user: null,
      theme: 'light',
      actions: {
        setUser: (user) => set({ user }),
        setTheme: (theme) => set({ theme })
      }
    }),
    { name: 'user-storage' }
  )
)

규칙: Persist는 필요한 상태만 저장

  • WHEN persist를 사용할 때
  • THEN partialize 옵션으로 저장할 상태를 명시 (actions는 제외)
  • WHY 불필요한 데이터 저장 방지 및 저장소 크기 최소화

CORRECT:

persist(
  (set) => ({ /* ... */ }),
  {
    name: 'app-settings',
    partialize: (state) => ({
      theme: state.theme,
      language: state.language
      // actions는 제외
    })
  }
)

파생 상태 규칙

규칙: 파생 상태는 컴포넌트에서 계산

  • WHEN 기존 상태를 조합하거나 필터링한 값이 필요할 때
  • THEN store에 저장하지 말고 컴포넌트에서 계산하거나 selector 함수로 분리
  • WHY store를 단순하게 유지하고 불필요한 상태 관리 제거

CORRECT (컴포넌트에서 계산):

const TodoList = () => {
  const { todos, filter } = useStore(
    state => ({ todos: state.todos, filter: state.filter }),
    shallow
  )
  
  const filteredTodos = todos.filter(todo => {
    if (filter === 'active') return !todo.completed
    if (filter === 'completed') return todo.completed
    return true
  })
  
  return <ul>{/* ... */}</ul>
}

CORRECT (selector 함수):

// store.js
export const selectFilteredTodos = (state) => {
  const { todos, filter } = state
  return todos.filter(todo => {
    if (filter === 'active') return !todo.completed
    if (filter === 'completed') return todo.completed
    return true
  })
}

// component
const filteredTodos = useStore(selectFilteredTodos)

INCORRECT (store에 파생 상태):

const useStore = create((set, get) => ({
  todos: [],
  filter: 'all',
  getFilteredTodos: () => { /* ... */ } // ❌
}))

React 외부 접근 규칙

규칙: React 외부에서 상태가 필요하면 subscribe 사용

  • WHEN utils, services 등 React 컴포넌트가 아닌 곳에서 store 상태를 사용할 때
  • THEN useStore.subscribe() 사용
  • WHY React 훅 없이도 상태 변화 감지 및 처리 가능

CORRECT:

// utils/analytics.js
export const setupAnalytics = () => {
  const unsubscribe = useStore.subscribe(
    (state) => state.user,
    (user, prevUser) => {
      if (user && !prevUser) {
        analytics.track('User Login', { userId: user.id })
      }
    }
  )
  
  return unsubscribe
}

비동기 작업 규칙

규칙: 비동기 액션은 loading/error 상태를 함께 관리

  • WHEN API 호출 등 비동기 작업을 수행할 때
  • THEN isLoading, error 상태를 함께 관리
  • WHY 로딩 상태와 에러 처리를 통한 UX 개선

CORRECT:

const useProductStore = create((set) => ({
  products: [],
  isLoading: false,
  error: null,
  
  actions: {
    fetchProducts: async () => {
      set({ isLoading: true, error: null })
      try {
        const response = await fetch('/api/products')
        const products = await response.json()
        set({ products, isLoading: false })
      } catch (error) {
        set({ error: error.message, isLoading: false })
      }
    }
  }
}))

규칙: 서버 상태는 react-query 사용 권장

  • WHEN 서버 데이터를 다룰 때
  • THEN Zustand는 클라이언트 상태(UI 상태)만 관리하고, 서버 상태는 react-query 사용
  • WHY loading/error 자동 처리, 캐싱, 재시도 등 서버 상태 관리에 최적화된 기능 제공

CORRECT:

// Zustand: 클라이언트 상태만
const useUIStore = create((set) => ({
  selectedProductId: null,
  filterCategory: 'all',
  actions: {
    setSelectedProduct: (id) => set({ selectedProductId: id }),
    setFilterCategory: (category) => set({ filterCategory: category })
  }
}))

// react-query: 서버 상태
const { data: products, isLoading, error } = useQuery({
  queryKey: ['products'],
  queryFn: async () => {
    const response = await fetch('/api/products')
    return response.json()
  }
})
