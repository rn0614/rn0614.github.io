Obsidian MCP 서버 응답 지연 및 Lock 문제 해결 가이드

배경/상황

Claude Desktop에서 Obsidian MCP를 npx로 실행하여 사용하던 중, 긴 콘텐츠 요청 후 서버가 응답하지 않는 문제가 발생했습니다.

초기 MCP 설정

{
  "obsidian": {
    "command": "npx",
    "args": [
      "-y",
      "obsidian-mcp",
      "C:\\work\\cursor\\blogging\\quartz\\content"
    ]
  }
}

문제 증상

1. 짧은 콘텐츠는 정상 작동

다음과 같은 간단한 노트 생성은 문제없이 동작했습니다:

{
  vault: "content",
  content: "# 재시작 테스트\\n\\nMCP 재연결 후 테스트입니다.\\n\\n정상 작동 확인!",
  filename: "restart-test.md"
}

2. 긴 콘텐츠 요청 후 서버 Lock

React Hooks 가이드처럼 긴 콘텐츠(특히 코드 블록 포함)를 요청하면:

  • 요청이 완료되지 않음
  • 응답 없음 (No result received from client-side tool execution)
  • 이후 모든 요청이 실패

3. Lock 이후 간단한 요청도 실패

긴 콘텐츠 요청 후에는:

  • 짧은 콘텐츠 요청도 실패
  • 볼트 목록 조회 같은 단순 작업도 실패
  • Claude Desktop 재부팅 전까지 복구 불가

문제 분석 과정

초기 가설: 백틱(`) 특수문자 문제

처음에는 코드 블록의 백틱이 JSON 직렬화 과정에서 문제를 일으킨다고 의심했습니다.

검증 결과: 백틱 없는 중간 길이 글에서도 동일한 멈춤 현상 발생

실제 원인 분석

1. JSON 직렬화 문제 가능성

긴 콘텐츠의 특수문자들이 JSON으로 전송될 때 제대로 이스케이프되지 않을 수 있습니다:

  • 백틱 (`)
  • 화살표 함수 (=>)
  • 중괄호, 대괄호 ({}, [])
  • 템플릿 리터럴

2. MCP 서버 응답 대기 상태 (Stuck)

긴 요청 처리 중 에러 발생 시:

  • 서버가 응답을 보내지 못하고 stuck 상태
  • 이후 요청이 큐에 쌓여서 처리 안 됨
  • 타임아웃도 발생하지 않음

3. 버퍼/타임아웃 제한

npx로 실행되는 프로세스가:

  • stdout 버퍼 한계에 도달
  • 또는 특정 타임아웃 설정 초과

npx 실행의 한계

npx -y obsidian-mcp <vault-path>

이 방식의 문제점:

  • 에러 로그 부재: stderr 출력을 볼 수 없음
  • 디버깅 불가: 실제 에러 원인 파악 불가
  • 커스터마이징 불가: 코드 수정이나 로깅 추가 불가

해결 방법

1단계: 로컬 빌드로 전환

# 저장소 클론
git clone https://github.com/StevenStavrakis/obsidian-mcp
cd obsidian-mcp

# 의존성 설치
npm install

# 빌드
npm run build

2단계: Claude Desktop 설정 변경

{
  "mcpServers": {
    "obsidian": {
      "command": "node",
      "args": [
        "C:\\work\\mcp\\obsidian-mcp\\build\\main.js",
        "C:\\work\\cursor\\blogging\\quartz\\content"
      ]
    }
  }
}

3단계: MCP Inspector로 디버깅

# MCP Inspector 실행
npx @modelcontextprotocol/inspector node ./build/main.js "C:\\work\\cursor\\blogging\\quartz\\content"

MCP Inspector를 사용하면:

  • 요청/응답을 실시간으로 확인
  • 어디서 에러가 발생하는지 정확히 파악
  • JSON 직렬화 문제 검증 가능

주요 인사이트

npx vs 로컬 빌드 비교

항목 npx (현재) 로컬 빌드
에러 로그 ❌ 안 보임 ✅ stderr로 확인 가능
디버깅 ❌ 불가 ✅ Inspector 사용 가능
수정 ❌ 불가 ✅ 코드 수정 가능
버전 관리 최신 자동 수동 업데이트
설정 간편함 ✅ 간단 약간 복잡

Lock 문제의 특징

  1. 재현 가능한 패턴: 긴 콘텐츠 → Lock → 재부팅 필요
  2. 누적 효과: 한 번 lock되면 모든 후속 요청 실패
  3. 재부팅으로 해결: MCP 서버 프로세스 재시작 필요

특수문자 문제 가능성

코드 블록 포함 시 특히 주의:

  • 백틱 3개 연속 사용
  • 템플릿 리터럴 내부 변수
  • JSON 문자열 내 따옴표 중첩

실용적 정보

임시 해결책 (로컬 빌드 전)

  1. 짧은 단위로 분할: 긴 콘텐츠를 여러 작은 노트로 나누기
  2. 코드 블록 최소화: 가능하면 코드 예시 줄이기
  3. 재부팅 주기적으로: lock 발생 시 즉시 재부팅

근본적 해결책

  1. 로컬 빌드 사용: 에러 로그를 확인할 수 있어야 함
  2. MCP Inspector 활용: 실시간 디버깅으로 원인 파악
  3. 코드 수정: 필요시 타임아웃이나 버퍼 크기 조정

추가 테스트 방법

특수문자 vs 길이 구분:

# 테스트 1: 특수문자 없는 긴 텍스트
Lorem ipsum... (1000+ words, no code blocks)

# 테스트 2: 짧은 코드 블록
```javascript
const x = 1;
```끝
  • 테스트 1 성공, 테스트 2 실패 → 특수문자 문제
  • 테스트 1 실패, 테스트 2 성공 → 길이 제한 문제
  • 둘 다 실패 → 복합 원인

참고/연결

관련 리소스

  • Obsidian MCP GitHub
  • MCP Inspector 문서
  • Claude Desktop 설정 파일 위치: %APPDATA%/Claude/claude_desktop_config.json

다음 단계

  1. 로컬 빌드 설정 완료
  2. MCP Inspector로 실제 에러 로그 확인
  3. 원인 파악 후 이슈 리포트 또는 코드 수정
  4. 안정적인 콘텐츠 길이 제한 파악

결론

npx 실행 방식은 간편하지만 디버깅이 불가능하다는 치명적인 단점이 있습니다. 긴 콘텐츠 요청 시 발생하는 lock 문제를 해결하려면 로컬 빌드로 전환하여 실제 에러 로그를 확인해야 합니다.

MCP 서버의 안정성을 확보하기 위해서는:

  • 에러 처리 개선
  • 타임아웃 설정 적절히 조정
  • 버퍼 크기 제한 확인
  • 요청 크기 제한 명시

이러한 디버깅 과정을 통해 MCP 서버를 더 안정적으로 운영할 수 있습니다.
